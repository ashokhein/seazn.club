# UX Audit — Console Org-Level Pages

Scope: `/o/[org]` home, `/o/[org]/c/new`, `/o/[org]/settings` (all tabs), `/o/[org]/settings/billing`, `/o/[org]/settings/payments`, `/directory` (players/clubs), `/import`.
Accounts: Riverside Sports Club (`org-7351fd0bed`, pro, owner `ashokhein+demo1@gmail.com`) and Northside Community Club (`org-ace12ac11d`, free, owner `ashokhein+demo2@gmail.com`). Password for both (local dev DB only): `AuditPass123!` — but login is magic-link only in the UI (no password field), see gotcha below.
Viewports: desktop 1440x900, mobile 390x844.

## Gotchas hit during this audit (for other forks / future sessions)
- Dev server threw `Module not found: '@formatjs/intl-localematcher'` on first boot — `npm install` at repo root fixed it (packages existed in package.json but not installed on disk). If you see a 500 on every page, run `npm install` at repo root, not just apps/web.
- Playwright MCP shared browser had a stale `SingletonLock` in `~/Library/Caches/ms-playwright-mcp/mcp-chrome-e83ee0b/` blocking new connections. Removed `SingletonLock`/`SingletonCookie`/`SingletonSocket` — fixed it. If parallel forks share this one browser/profile, expect navigation race conditions (saw one mid-navigation 405 chrome-error blip that resolved on retry).
- `browser_click`/`browser_type` `target` must be the bare ref string (e.g. `f4e103`), NOT `ref=f4e103` — the latter throws "Unknown engine ref".
- Login is passwordless magic-link only (no password field in `/login` UI despite `users.password_hash` column existing). To get a session for scripted audits: submit email on `/login`, then read the token straight from local Postgres `seazn_club.login_links` (`select l.token from login_links l join users u on u.id=l.user_id where u.email=... order by l.created_at desc limit 1`), then navigate to `/magic-link?token=<token>`.
- Screenshots via `browser_take_screenshot` save relative to the Next.js dev server's cwd, which resolved to repo root (`/Users/ashokhein/github/seazn.club/<filename>.png`), not `apps/web` and not the MCP's own `.playwright-mcp/` dir.
- The Riverside demo account's console renders **entirely in French** (locale=fr) — not something this audit changed. Worth confirming with the team whether that's an intentional per-account locale preference or leftover test-account state; either way, some strings on this account leak English regardless (see finding below), which is a real bug independent of which locale is active.

### [high] `/o/[org]` (org home, filled state) — mobile 390px — floating help/chat bubble covers competition card content
**What I saw:** The fixed-position circular "N" help/chat launcher bottom-left sits ON TOP of the competition card list when there are enough cards to scroll near it — it visibly covers the medal emoji and clips the "Win" off "Winter Indoor Open" title, plus overlaps part of the card border. See `screenshots/02-org-home-filled-mobile.png`. This isn't just a z-index nuisance, it hides real content (part of a competition's name).
**Fix prompt:** The floating help/chat FAB (bottom-left, circular "N" icon) needs a higher z-index AND its own reserved margin so page content doesn't render underneath it, or better: give the competitions list container `padding-bottom` / `scroll-margin` sized to the FAB's height + safe margin on mobile so cards never sit behind it. Also consider only showing the FAB after the user scrolls past the header, or moving it to bottom-right (away from primary content flow) to match common mobile FAB conventions. Find the FAB component (likely a `HelpButton`/`SupportWidget` mounted globally in the console layout) and its CSS positioning.

### [medium] `/o/[org]` (org home, filled state) — desktop — English words leak into French-localized UI
**What I saw:** With the Riverside account's locale set to French, the whole competitions list renders in French (Compétitions, Inscriptions ouvertes, Brouillon, Terminé, etc.) except the "Next fixture" line on each competition card, which reads e.g. `Next: Now: Harbor CC vs Summit CC · Court 3 · Tue 21 Jul, …` — both "Next:" and "Now:" stay in English, and the date (`Tue 21 Jul`) is also un-localized, inside an otherwise fully-French card.
**Fix prompt:** Find the org-home competition-card component (likely in `o/[orgSlug]/page.tsx` or a `CompetitionCard`/similar component it renders) and locate the "Next: Now: X vs Y · Court N · date" string. It's building this from raw English literals instead of the `ui` message catalog (`msgFor`/`useMsg`) used elsewhere on this page. Route it through the same i18n catalog as the rest of the page, and format the date via the locale-aware date formatter already used elsewhere in the app (see `[[project_v5_i18n_modernized]]` for the established pattern) instead of a raw `Tue 21 Jul` string.

### [low] `/o/[org]` (org home, filled state) — desktop — truncated "Next" line has no title/tooltip
**What I saw:** The "Next: …" line on competition cards is single-line truncated with `…` when the fixture description + court + date is long (e.g. Autumn Cup 2026 card cuts off after "Tue 21 Jul, …"). There's no `title` attribute or affordance to see the full text.
**Fix prompt:** Add a `title="…full text…"` attribute (or a tooltip) to the truncated "Next" paragraph so the full fixture description/time is available on hover, since it's meaningful scheduling info being hidden by ellipsis.

### [high] `/o/[org]/c/new` (new competition wizard) — desktop — form is bilingual English/French on the same screen
**What I saw:** With the account locale set to French, every other console page audited so far renders fully in French — but this wizard's field labels, breadcrumb, radio options and buttons ("Name", "Description", "Who can see this", "Private/Link only/Public", "Starts on"/"Ends on", "Brand accent colour (Pro branding)", "Cancel"/"Create competition") are all in English, while ONE card in the middle ("Mettre en avant sur seazn.club" — the featured-on-discover opt-in, including its full explanatory paragraph and the orange warning line) is fully translated to French. Screenshot `screenshots/02-comp-new-step1-desktop.png`.
**Fix prompt:** This component (`CompetitionWizard`, `o/[orgSlug]/c/new/page.tsx`) is missing from the i18n catalog coverage that the rest of the console got — most of its copy is hardcoded English literals while the "feature on seazn.club" sub-block alone was wired through the `ui` message catalog. Route the rest of the wizard's strings (name/description labels, visibility radio group, date labels, brand colour label, buttons) through the same `msgFor`/`useMsg()` pattern, and add the missing keys to `fr`/`es`/`nl` catalogs so this doesn't read as half-finished.

### [high] `/o/[org]/c/new` — mobile 390px — floating help bubble sits directly over the "feature on seazn.club" checkbox, likely blocking the tap target
**What I saw:** Same FAB as the org-home finding above, but here it's worse: the circular "N" button overlaps the actual checkbox input for "Mettre en avant sur seazn.club", not just decorative background. If the FAB's hit area extends over the checkbox (likely, since it's a real button), users can't check that box on mobile without first dismissing/scrolling past the FAB. Screenshot `screenshots/02-comp-new-mobile.png`.
**Fix prompt:** Same root fix as the org-home FAB overlap (reserve space / reposition / raise content z-index), but treat this instance as higher priority since it's blocking an interactive control, not just visually covering text. Test after the fix that the checkbox is actually clickable via tap at this viewport width.

### [medium] `/o/[org]/settings` — desktop — bilingual English/French leak is systemic across several tabs, not just one
**What I saw:** With the account locale set to French, the settings shell (sidebar nav, breadcrumb-adjacent "Paramètres" header, tab labels) is French, but tab BODIES are inconsistently translated:
  - `?tab=organization`: mostly English body ("Organization name", "Save"), except the "Mettre en avant"-style blocks which are French. Also "Enregistrer À propos" button reads like two independently-translated fragments concatenated ("Enregistrer" + "À propos") rather than one natural French phrase.
  - `?tab=sponsors`: fully French, looks correct (good reference example — see `screenshots/02-settings-sponsors-desktop.png`).
  - `?tab=team`: entirely English body ("MEMBERS", "Owner", "INVITE BY EMAIL", "Send invite", "INVITE BY LINK", "Create link") except the "Équipe" heading. See `screenshots/02-settings-team-desktop.png`.
  - `?tab=api`: same pattern — "Key name" placeholder, "Create key", "What can this key do?", empty-state copy, "Full reference, guides..." all English; only the 3 scope names (Lecture/Score/Gestion) and their descriptions are French. See `screenshots/02-settings-api-desktop.png`.
  - `/settings/billing` (Plan & billing): "Downgrade to Community" button is English, breadcrumb "Plan & billing" is English, rest of page (Forfait actuel/Utilisation/Compétitions actives etc.) is French. See `screenshots/02-settings-billing-desktop.png`.
  - Breadcrumb second segment always shows "Settings" in English regardless of locale.
**Fix prompt:** The Sponsors tab shows the correct pattern (fully wired to the `ui` message catalog) — use it as the template. Audit `o/[orgSlug]/settings/page.tsx` and its `organization`/`team` tab-body components/sub-components for hardcoded English literals and route them through `msgFor`/`useMsg()` the same way Sponsors was done, adding the missing keys to `fr`/`es`/`nl`. Also fix the breadcrumb's "Settings" segment to pull from the locale catalog. For the concatenated button label on the About-page save button, add one full translated string per save button (e.g. `settings.about.save`) instead of templating a generic "Save" + a raw section name — French word order won't match English concatenation.

### [low] `/o/[org]/settings/billing` — desktop — no payment method / invoice history section visible for an active Pro plan
**What I saw:** Billing page for an active-Pro org shows only "Current plan" + "Usage" cards — no payment method on file, no invoice/receipt history, no "Manage in Stripe" link. Might be expected for local dev without real Stripe keys (3 console errors + 1 warning fired on this page, likely Stripe.js failing to init without a real publishable key), but worth a sanity check against a real staging org to confirm the payment-method/invoice sections do render when Stripe is properly configured.
**Fix prompt:** Not a confirmed bug — flagging for verification. If this is dev-only due to missing Stripe test keys, no action needed; if it also looks like this on staging for a genuinely active-Pro org, the payment-method/invoice history sections are missing/broken and need investigating (check `o/[orgSlug]/settings/billing/page.tsx`).

### [high] `/o/[org]/settings` — mobile 390px — tab bar disappears entirely on scroll, no way to switch tabs without scrolling back to top
**What I saw:** On mobile the settings sidebar becomes a horizontally-scrollable sticky tab row ("Organisation / Sponsors / Équipe / API…") — good idea. But it's `position: sticky; top: 0` with the SAME `top: 0` as the page header above it (verified via DOM inspection: header is `.app-gantry.sticky.top-0.z-20` h=58px, tab row is `.scroll-x.sticky.top-0.z-30` h=52px). Both competing for `top: 0` means the tab row doesn't actually stay pinned below the header as scroll proceeds — once you scroll down ~120px, the tab row scrolls away completely and only the header remains. Verified with a scroll-then-screenshot test (`screenshots/02-settings-mobile-scrolled-sticky-check.png`): after scrolling to y=400, the tab row is gone, no way to switch settings tabs without scrolling all the way back to the top of the page. (An initial full-page screenshot appeared to show the header duplicated mid-page — that was a Playwright full-page stitching artifact from the sticky header, not a real bug; confirmed via DOM query that only one header exists.)
**Fix prompt:** Find the settings tab-row component (mobile sticky tab nav in `o/[orgSlug]/settings/page.tsx` or its nav sub-component, class `scroll-x sticky top-0 z-30 ... md:static`). Change its sticky offset to `top: 58px` (or `top: [header-height]` via a CSS var so it tracks the actual header height) instead of `top: 0`, so it stays pinned directly below the header rather than competing for the same position and scrolling out of view.

### [low] `/o/[org]/settings/billing` — desktop — Pro-tier billing page is noticeably sparser than the free-tier one
**What I saw:** Comparing the two accounts: the Community (free) billing page shows a full plan-comparison card (Community vs Pro feature lists, trial CTA, annual/monthly toggle). The Pro billing page (`screenshots/02-settings-billing-desktop.png`, Riverside) shows only "Current plan" + "Usage" + a single "Downgrade to Community" button — no visible payment method, invoice history, or plan-comparison-in-reverse (what you'd lose by downgrading, beyond the one button). It reads like an unfinished counterpart to the free page.
**Fix prompt:** Not urgent, but worth a design pass: give Pro-tier billing parity with the free-tier page's polish — show payment method on file, link to invoice history, and consider showing what downgrading removes (mirroring the free page's ✓/✗ feature comparison) so the decision is as clear in both directions.

Card grid renders cleanly, status badges (En direct/Inscriptions ouvertes/Brouillon/Terminé) are visually distinct, progress bars show correct completion ratios, sport emoji + gradient card header look consistent across cards.

### [high] `/o/[org]` (org home, EMPTY state) — desktop — product tour dialog fully covers the empty-state CTA it's supposed to explain
**What I saw:** On a brand-new org with zero competitions, the "Welcome to Seazn Club" product tour dialog opens centered on load, directly on top of the empty-state message ("No competitions yet — create one in about two minutes.") and its "Create your first competition" button — both are completely hidden behind the modal's white card. See screenshot `screenshots/02-org-home-empty-desktop-tour-cookie-stack.png`.
**Fix prompt:** Either (a) don't auto-open the product tour dialog on top of the zero-state screen — the empty-state CTA itself already does the job the tour dialog is trying to do here, so suppress the tour on this specific route/condition, or (b) reposition the tour dialog so it doesn't occlude the primary CTA (e.g. anchor it to a corner or add a spotlight/cutout around the CTA instead of a centered modal). Likely file: whatever renders `dialog "Product tour"` — search for the tour trigger logic (probably a `ProductTour`/`Tour` component gated by a first-login flag) and the org-home empty-state condition (`competitions.length === 0` check in `o/[orgSlug]/page.tsx`).

### [low] Global — desktop — cookie-consent banner stacks directly on a floating help/chat bubble
**What I saw:** Same screenshot (`screenshots/02-org-home-empty-desktop-tour-cookie-stack.png`): the bottom-left cookie consent banner sits with its bottom edge almost touching a circular "N" floating action button (looks like a chat/help launcher), and at narrower viewports these will likely overlap.
**Fix prompt:** Give the cookie banner a bottom offset that accounts for the floating action button's height + margin (e.g. `bottom: calc(<fab-height> + <fab-margin> + <gap>)`), or move the FAB to the opposite corner while the cookie banner is visible. Confirm on mobile too (checked below).

## Summary

**Pages/tabs/modals checked:** org home (empty + filled, desktop + mobile), new-competition wizard (desktop + mobile), org settings tabs — Organisation/Sponsors/Team/API/Billing/Payments (desktop, + Organisation on mobile incl. scroll behavior), Directory Players + Clubs tabs (desktop), Import wizard pre-upload state (desktop). Cross-checked Sponsors and Billing tabs on both pro (Riverside) and free (Northside) accounts.

**Not reached, out of budget for this pass:** competition-level settings/schedule/upgrade pages, division-level pages (entrants/fixtures/standings/stats/schedule tabs — covered by a separate audit per the task split), Registrations panel modals, Import wizard's column-mapping/preview steps (needs an actual file upload), mobile views of Sponsors/Team/API/Billing/Payments/Directory/Import (only Org home + New competition + Organisation-settings got the mobile pass; same FAB-overlap and sticky-tab-bar bugs found there likely recur on every console page since both bugs are in shared layout chrome, not page-specific components).

**Severity counts:** 5 high, 3 medium, 4 low, 1 unconfirmed/needs-verification, 4 OK notes.

**Top 3 priority fixes:**
1. Settings tab bar disappears entirely on mobile scroll (sticky `top:0` collision with header) — blocks navigation, affects every settings tab.
2. Floating help/chat FAB overlaps content and, on the new-competition wizard, an actual checkbox tap target — affects mobile across the whole console (shared global component).
3. Product tour modal fully covers the org-home empty-state CTA on first login — breaks the exact first-time-user flow the tour exists to support.

Close behind: the bilingual English/French leak is systemic (new-competition wizard, org/team/api settings tab bodies, billing button) — low visual severity per instance but adds up to a "half-finished" impression across most of the org-level console for non-English locales.
